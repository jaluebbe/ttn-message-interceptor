LoRaWAN – what is it?

  • Long Range Wide Area Network
  • Operates at 868 MHz in Europe
  • Long range, low power, low data rate
  • Encrypted transmission
LoRaWAN operates at 868 MHz in Europe. Long range means up to a few kilometres depending on the environment. Designed for low-power, low-data-rate applications with encrypted transmission.

Typical sensor nodes

Temperature
Water level
GPS location
Battery life: temperature or water level sensors may last years. A GPS tracker may only last days, depending on the update rate.

LoRaWAN architecture

  • Gateway forwards raw frames to The Things Stack Sandbox
  • Decryption and decoding happen on the Network Server
  • Cloud dependency is structural
The gateway forwards raw frames and has no knowledge of the payload content. Decryption happens in the cloud. That structural dependency is what we address. Note: "TTN" on slides = The Things Stack Sandbox (TTSS), the free non-commercial tier.

LoRaWAN data rate & limits

  • Up to 5.5 kbit/s – roughly comparable to a fax machine 📠 (still indispensable in Germany)
  • Legal duty cycle limit: 1% → max. 36 s/h on air
  • TTN fair use policy: max. 30 s airtime per day
Airtime for a 21-byte frame (8 byte payload + 13 byte header): SF7: ~56 ms → ~535 messages/day SF9: ~206 ms → ~145 messages/day SF12: ~1.5 s → ~20 messages/day

A LoRaWAN message

The gateway receives an encrypted LoRaWAN frame:

MHDR DevAddr FCnt FPort frmPayload MIC
40 d4c3b2a1 0200 01 2c46e592c0b8c064 24fc865e
A1B2C3D4 2

The gateway forwards the frame to TTN as-is.
Everything visible except the frame payload.

All fields except frmPayload are unencrypted and readable without any keys. The gateway forwards the frame unchanged to TTN. DevAddr and FCnt are transmitted little-endian on the wire (d4c3b2a1 → A1B2C3D4, 0200 → 2).

From gateway to application

  • TTN decrypts the payload and routes it by application ID and device ID
  • Each device has a JavaScript decoder in TTN
// TTN decoder for my_application / my_device
function decodeUplink(input) {
  return {
    data:     { bytes: input.bytes },
    warnings: [],
    errors:   []
  };
}
module.exports = { decodeUplink };
TTN decrypts the payload using the session keys stored on the Network Server. The decoded result is routed to the correct application using application ID and device ID. Each device can have its own JavaScript decoder – this one simply passes through the raw bytes.

Sensor online, internet offline

  • Water is rising
  • Internet is gone
  • All data disappears into nowhere
The sensor keeps transmitting – water level data, reliably sent. But without internet, nobody receives it.

The hybrid approach

  • Act as a TTN gateway
  • Decode all own device messages locally
  • No changes to devices
  • No changes to TTN

Hardware: Raspberry Pi with LoRaWAN concentrator board and antenna, LTE modem/stick for internet, GPS module for connectivity and system time (the Pi does not retain time after a power outage).

UDP stream duplication

lora_pkt_fwd (SPI)
      |  UDP :1700
 gwmp_mux.py  # duplicates uplink frames
      |
      +---> TTN cloud :1700
      +---> localhost :1701
                |
      message_collector -> Redis -> message_handler

Every frame is also captured locally for decryption.

gwmp_mux.py listens on UDP port 1700 and forwards every uplink frame to both TTN and our local collector. Local decryption only works for known devices – session keys must have been fetched from TTN at least once before the outage.

LoRaWAN encryption

  • Each device has its own session keys (AppSKey, NwkSKey)
  • Without the keys, the payload is unreadable
  • Keys are stored on the Network Server – fetch them via TTN API
Two keys per device: AppSKey for payload encryption, NwkSKey for integrity check (MIC). Both are stored on the TTN Network Server and fetched once via the API. Without them, the ciphertext in the frame is unreadable.

Fetching session keys from TTN

for app_id in application_ids:
    for device in list_application_devices(session, app_id):

        store_device_session_in_db(
            dev_eui    = net_data["ids"]["dev_eui"],
            dev_addr   = net_data["session"]["dev_addr"],
            app_s_key  = app_data["session"]["keys"]["app_s_key"]["key"],
            nwk_s_key  = net_data["session"]["keys"]["f_nwk_s_int_key"]["key"],

Two separate API calls per device: NwkSKey comes from the Network Server, AppSKey from the Application Server. Both are stored in SQLite and used for offline decryption. The script runs daily via cron to keep keys up to date.

Decryption at the gateway

# message_processor.py
raw_hex = "40" "d4c3b2a1" "00" "0200" "01" "2c46e592c0b8c064" "24fc865e"
#          MH   DevAddr    FC   FCnt   FP   frmPayload         MIC

info = requests.post(f"{BASE_URL}/info/hex",
    json={"payload": raw_hex}).json()
# --> devAddr, fPort, fCnt, frmPayload (encrypted)

session = get_latest_session_by_dev_addr(info["devAddr"])
# --> application_id, device_id, app_s_key, nwk_s_key

decrypted = requests.post(f"{BASE_URL}/decrypt/hex", json={
    "payload":   raw_hex,
    "app_s_key": session["app_s_key"],  # payload decryption
    "nwk_s_key": session["nwk_s_key"],  # MIC verification
}).json()
# --> "f09f918bf09f8c8d"
lora-packet handles MIC verification (NwkSKey) and AES decryption (AppSKey). DevAddr is little-endian on the wire – lora-packet converts it to big-endian for display.

The payload decoder problem

  • Device payload decoders exist as JavaScript only
  • Python has no access to JS libraries directly
  • Options: rewrite in Python, call subprocess, or…
Rewriting decoders in Python was attempted – feasible for simple decoders, but too complex for real-world device decoders.

Node.js HTTP service

// nodejs/decoders_api.js
const express = require("express");
const lora_packet = require("lora-packet");

const app = express();
app.use(express.json());

// ... extractMessageInfo, decryptPayload, decodePayload

app.post("/info/hex",    (req, res) => extractMessageInfo(req, res, "hex"));
app.post("/decrypt/hex", (req, res) => decryptPayload(req, res, "hex"));
app.post("/decode/hex",  (req, res) => decodePayload(req, res, "hex"));

app.listen(3000);
Explicit JSON middleware (express.json()) is required to parse request bodies – FastAPI handles this automatically. lora-packet provides the LoRaWAN crypto operations. The three routes serve the three steps in message_processor.py. Why not subprocess? Node.js starts once and stays running; errors surface as HTTP status codes. Rewriting decoders in Python was attempted – feasible for simple decoders, but too complex for real-world device decoders.

Payload decoding

function decodeUplink(input) {
  return {
    data:     { text: Buffer.from(input.bytes).toString("utf8") },
    warnings: [],
    errors:   []
  };
}
module.exports = { decodeUplink };
decoded = requests.post(f"{BASE_URL}/decode/hex", json={
    "payload":     decrypted,                  # "f09f918bf09f8c8d"
    "application": session["application_id"],  # "demo"
    "device":      session["device_id"],       # "hello_world"
    "fPort":       info["fPort"],              # 1
}).json()
The JS decoder is minimal – just converts raw bytes to UTF-8. Python receives a clean JSON response. The actual result will be revealed on the last slide. The Python equivalent: bytes.fromhex("f09f918bf09f8c8d").decode("utf-8") # 👋🌍

Local data storage & API

  • Decoded messages stored in SQLite
  • FastAPI serves the data on port 8080
  • Data stays accessible offline
The API is read-only – the gateway writes to SQLite, the API only reads from it.
The API provides both the latest values and time series data for each sensor. Messages received before session keys were fetched are stored and reprocessed once keys become available – they are not lost permanently.
Red: stopped – no internet connection to TTN. Blue: frozen – last known state, no updates. White: fully operational – local processing continues. Devices must have connected to TTN at least once before a blackout. Register devices before deployment, not during an incident. OTAA re-join tip (e.g. Dragino PS-LB): AT+DDETECT=0 disables automatic re-join – device keeps its session indefinitely. For a manual re-join, a physical reset button press suffices.

Don't rewrite what already exists – orchestrate it.

  • lora-packet (JS) for LoRaWAN decryption
  • Manufacturer decoders (JS) for payloads
  • Python for orchestration and data storage

HTTP makes language boundaries irrelevant.

lora-packet and the device decoders already exist and are well-maintained. Python connects the pieces.
