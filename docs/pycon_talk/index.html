LoRaWAN – what is it?

  • Long Range Wide Area Network
  • Operates at 868 MHz in Europe
  • Long range, low power, low data rate
  • Encrypted transmission
LoRaWAN operates at 868 MHz in Europe. Designed for long-range, low-power, low-data-rate applications. All messages are encrypted.

Typical sensor nodes

Temperature
Water level
GPS location
Battery life: temperature or water level sensors may last years. A GPS tracker may only last days, depending on the update rate.

LoRaWAN architecture

  • Gateway forwards raw frames to The Things Stack Sandbox
  • Decryption and decoding happen on the Network Server
  • Cloud dependency is structural
The gateway forwards raw frames and has no knowledge of the payload content. Decryption happens in the cloud. That structural dependency is what we address. Note: "TTN" on slides = The Things Stack Sandbox (TTSS), the free non-commercial tier.

LoRaWAN data rate & limits

  • Up to 5.5 kbit/s – roughly comparable to a fax machine 📠 (still indispensable in Germany)
  • Legal duty cycle limit: 1% → max. 36 s/h on air
  • TTN fair use policy: max. 30 s airtime per day
Airtime for a 21-byte frame (8 byte payload + 13 byte header): SF7: ~56 ms → ~535 messages/day SF9: ~206 ms → ~145 messages/day SF12: ~1.5 s → ~20 messages/day

Sensor online, internet offline

  • Water is rising
  • Internet is gone
  • All data disappears into nowhere
The sensor keeps transmitting – water level data, reliably sent. But without internet, nobody receives it.

The hybrid approach

  • Act as a TTN gateway
  • Decode all own device messages locally
  • No changes to devices
  • No changes to TTN

Tracing a message

The gateway receives an encrypted LoRaWAN frame:

MHDR DevAddr FCnt FPort FRMPayload MIC
40 A1B2C3D4 0002 01 2c46e592c0b8c0642 4fc865e

Our interceptor picks this up in parallel to TTN.
Everything visible except the payload.

The frame format is standardized – DevAddr, FCnt, FPort, MHDR (MAC Header) and MIC (Message Integrity Code) are all unencrypted and can be read without any keys. Everything visible except the payload. Our interceptor captures the frame alongside TTN and will decode it locally.

LoRaWAN encryption

  • Each device has its own session keys (AppSKey, NwkSKey)
  • Without the keys, the payload is unreadable
  • Keys are stored on the Network Server – fetch them via TTN API
Two keys per device: AppSKey for payload encryption, NwkSKey for integrity check (MIC). Both are stored on the TTN Network Server and fetched once via the API. Without them, the ciphertext in the frame is unreadable.

Fetching session keys from TTN

# request_ttn_devices.py (simplified)
for app_id in application_ids:
    for device in list_application_devices(session, app_id):
        net_data = request_device_keys(session, app_id, device["device_id"])
        app_data = request_app_device_data(session, app_id, device["device_id"])
        store_device_session_in_db(
            dev_eui    = net_data["ids"]["dev_eui"],
            dev_addr   = net_data["session"]["dev_addr"],
            app_s_key  = app_data["session"]["keys"]["app_s_key"]["key"],
            nwk_s_key  = net_data["session"]["keys"]["f_nwk_s_int_key"]["key"],
            ...
        )
Two separate API calls per device: NwkSKey comes from the Network Server, AppSKey from the Application Server. Both are stored in SQLite and used for offline decryption. The script runs daily via cron to keep keys up to date.

What about messages before the first key fetch?

  • Undecoded messages are stored and reprocessed once keys are available
  • Requires at least one successful TTN connection per device
  • Configure devices to avoid unnecessary re-joins
reprocess_messages.py retries decoding – messages received before the first key fetch are not lost permanently. Key limitation: devices must have connected to TTN at least once. In practice: register devices before deployment, not during an incident. Brand new devices in a blackout scenario cannot onboard at all. OTAA re-join tip (e.g. Dragino PS-LB): AT+DDETECT=0 disables automatic re-join – device keeps its session indefinitely. For a manual re-join after a longer outage, a physical reset button press at the device is sufficient.

The Node.js HTTP service uses the lora-packet library for MIC verification and AES decryption.

The payload decoder problem

  • Device payload decoders exist as JavaScript only
  • Python has no access to JS libraries directly
  • Options: rewrite in Python, call subprocess, or…
Rewriting decoders in Python was attempted – feasible for simple decoders, but too complex for real-world device decoders.

Node.js HTTP service

// nodejs/decoders_api.js  (key routes)
app.post("/info/hex",    (req, res) => extractMessageInfo(req, res, "hex"));
app.post("/decrypt/hex", (req, res) => decryptPayload(req, res, "hex"));
app.post("/decode/hex",  (req, res) => decodePayload(req, res, "hex"));
// decrypt uses lora-packet library
const packet = lora_packet.fromWire(Buffer.from(payload, "hex"));
lora_packet.verifyMIC(packet, NwkSKey);  // integrity check
const decrypted = lora_packet.decrypt(packet, AppSKey, NwkSKey);

Why HTTP and not subprocess?

  • Python and Node.js run independently
  • Errors surface as HTTP status codes
  • Node.js starts once, not per message

Local data storage & API

  • Decoded messages stored in SQLite
  • FastAPI serves the data on port 8080
  • Data stays accessible offline
The API is read-only – the gateway writes to SQLite, the API only reads from it.
The API provides both the latest values and time series data for each sensor.

Don't rewrite what already exists – orchestrate it.

  • lora-packet (JS) for LoRaWAN decryption
  • Manufacturer decoders (JS) for payloads
  • Python for orchestration and data storage

HTTP makes language boundaries irrelevant.

lora-packet and the device decoders already exist and are well-maintained. Python connects the pieces.
