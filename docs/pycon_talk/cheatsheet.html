01 Offline Fallback for a Mobile LoRaWAN Gateway

Thanks for the introduction and welcome to my talk.

02 lorawan_illustration_sensors_only.svg

At first I would like to mention some typical use cases of LoRaWAN sensors. Imagine a sunny day and you consider to go swimming – it would be nice to know the water temperature of the nearby lake. Or the opposite: it is raining a lot and you are afraid the water level of the river is threatening your home. Or you arranged a shipment and you would like to know where your assets are.

03 Typical sensor nodes

Let's take a look at these LoRaWAN sensors, which are called nodes. The battery life of temperature or water level sensors may last for several years and can be extended further with a small solar panel. In a GPS tracker like this one in my hand, the battery lasts just days or weeks depending on the update rate.

04 lorawan_illustration_normal_operation.svg

The messages sent by these sensor nodes are received by nearby gateways. The gateways forward the data to the network server where it could be picked up by the end user's application.

05 LoRaWAN – what is it?

But what exactly is LoRaWAN? LoRaWAN stands for Long Range Wide Area Network – where long range means up to a few kilometres depending on the environment. In Europe it is operated at 868 MHz. The power consumption is low, but so is the data rate. All transmissions are encrypted.

06 LoRaWAN data rate & limits

The data rate of LoRaWAN is up to 5.5 kilobits per second – comparable to a fax machine.

However, we are not limited by the availability of thermopaper but by legal requirements – the German Bundesnetzagentur enforces a 1% duty cycle limit.

In addition to that, there is the fair use policy of TTN allowing a maximum of 30 seconds airtime per day. But what does this mean for us? A typical LoRaWAN data frame has about 200 ms airtime, which means about 150 messages per day – or one every 5 minutes.

07 LoRaWAN architecture

Let's take a closer look at the gateway. It receives data frames from all nearby sensor nodes – not only ours – and forwards them to the network server. We are using The Things Stack Sandbox, the free tier of The Things Network for non-commercial use, which I'll refer to as TTN from now on. Decryption and decoding happen in the cloud, which means our operation relies on a working internet connection.

08 From gateway to application

On the network server, the message is matched by its device address to the corresponding session – which holds the keys to decrypt the payload. This also reveals the application ID and device ID, which are not publicly visible in the raw frame. Based on these IDs, the decrypted payload is passed to a device-specific JavaScript decoder stored in TTN. For a water level sensor this is where raw bytes get converted into actual centimetres. Here you see a minimal example that simply passes through the raw bytes.

09 Sensor online, internet offline

So far so good – as long as the internet connection is working. But imagine a water level sensor like this one, permanently deployed and especially critical during a flood event. The flood disrupts the internet connection. We can no longer observe the water level. What should we do?

10 lorawan_illustration_blackout_operation.svg

For this purpose I am using a Raspberry Pi based gateway which presents the received sensor data directly to the user. In the following I will explain all the steps of my journey.

11 The hybrid approach

My solution is the hybrid gateway – a Raspberry Pi with a LoRaWAN concentrator board, antenna, GPS module for accurate system time, and an LTE modem for internet connectivity. Under normal operation it acts as a standard TTN gateway, forwarding sensor data to the cloud. But at the same time it fetches data back from TTN that we need to decode messages locally in case of an outage. And it serves the decoded data directly to the user.

12 lorawan_illustration_full_operation.svg

Here you can see how all the pieces fit together – sensors, the two standard gateways, our Raspberry Pi gateway, the TTN cloud, and the end user. Under normal operation everything works in parallel.

13 UDP stream duplication

The LoRaWAN packet forwarder interacts with the LoRaWAN hardware, forwarding received data frames via UDP to the network server. This is where we start to intercept. Our multiplexer duplicates these frames – they are still forwarded to TTN if available, but also to our message collector service running on the Raspberry Pi. The messages are put into a Redis pipeline where they are picked up by our message handler to be processed and stored locally.

14 A LoRaWAN message

This is the raw LoRaWAN frame as received by the gateway. It consists of the message header, device address, frame control, frame counter, frame port, the encrypted payload, and a MIC – the message integrity code. The message type 0x40 indicates an unconfirmed uplink. Everything except the payload is readable without any keys. The second row shows the interpreted values – DevAddr and FCnt are transmitted little-endian on the wire. But there is still something missing to read the encrypted payload…

15 LoRaWAN encryption

To decrypt the payload locally, we need the session keys. The Network Session Key allows us to verify the integrity of the message, while the Application Session Key is required to actually decrypt the payload. We need to fetch these keys from the TTN cloud and cache them locally – while the internet connection is still available.

16 Fetching session keys from TTN

We need to perform two API requests per device to obtain both session keys as well as the individual payload decoder. If you have many devices, be mindful of the TTN API quota limits. All of this is stored in a local SQLite database.

17 Python meets JavaScript

For the next step we cannot continue with Python. The lora-packet library for decryption is written in JavaScript, and device decoders exist as JavaScript only. Rewriting is not an option when there are many different complex decoders. Therefore I encapsulate the JS code behind an HTTP API written in Node.js.

18 Node.js HTTP service

And this is what it looks like. I am using the Express framework, which looks somewhat similar to a simple FastAPI setup. We include the lora-packet library and a middleware to parse JSON request bodies. Let's skip the implementation details and focus on the three API endpoints representing the processing steps of our message handler – hosted on port 3000. You may ask: why an HTTP service instead of subprocess calls to Node.js from Python? The service starts once and stays running – not once per message – and errors surface as HTTP status codes, making them easier to handle.

19 Decryption at the gateway

Now we finally see Python code again. We start with the raw data frame you saw earlier. We push it to the info endpoint and obtain the device address along with other parameters. With the device address we look up our session database for the application ID, device ID, and session keys. This allows us to call the decryption endpoint – sending the full message and the session keys. Decryption requires not just the encrypted payload but also other frame parameters like the frame counter. Finally we have the decrypted payload – but we still don't know what it means.

20 Payload decoding

For this step we call the decode endpoint with application "demo" and device ID "hello_world". This allows us to look up the device-specific decoder, which in our case simply converts the bytes to a Unicode string. I will reveal what it contains on the last slide.

21 Local data storage & API

The decoded messages are stored in a local SQLite database. A read-only FastAPI service on port 8080 provides access to both the latest sensor values and historical time series data. During an internet outage, everything keeps running – the data stays accessible locally.

22 Data flow (normal operation)

Let me summarize the full data flow. On the left, request_ttn_devices fetches the session keys. On the right, ttn_storage_fetcher mirrors messages from TTN storage. Both write into SQLite. In the center, the message_handler processes incoming frames. reprocess_messages retries any messages that couldn't be decoded immediately due to missing session information. The REST API reads from both tables and serves the data to the client.

23 Data flow (internet outage)

During an outage, the red components stop working – we can no longer fetch new session keys or mirror TTN storage messages. The blue components are frozen at their last known state. But the white components keep running: gateway messages are still received and served via the REST API.
