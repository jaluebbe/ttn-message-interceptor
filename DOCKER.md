# Docker Deployment

This describes running the LoRaWAN processing pipeline in Docker. It receives
messages from one or more standalone Semtech-compatible gateways forwarding
UDP to port 1700.

## Architecture

```
Standalone LoRaWAN gateway(s)
        │ UDP :1700
lorawan-collector       receives GWMP UDP, publishes raw packets to Redis
        │
   Redis pub/sub
        │
lorawan-handler         decrypts + decodes → messages.db (SQLite)

lorawan-decoder         Node.js sidecar, called by lorawan-handler via HTTP
lorawan-tools           one-shot container for cron jobs and manual tasks
```

The `ttn_storage_fetcher.py` and `reprocess_messages.py` scripts run as
one-shot containers via cron rather than as persistent services.

## Directory layout on the host

```
docker-compose.yml
.env
data/                   mounted into lorawan-handler and lorawan-tools
  messages.db           created automatically on first run
  ttn_device_sessions.db  populate manually before first run (see below)
decoders/               generated by device_database.py, read by lorawan-decoder
```

## Setup

### 1. Clone the repository

```bash
git clone https://github.com/jaluebbe/ttn-message-interceptor.git github/ttn-message-interceptor
```

### 2. Create the `.env` file

```bash
cat > .env <<EOF
TTN_TOKEN=your_ttn_api_token_here
TTN_STORAGE_APPLICATIONS=my-app-1,my-app-2
EOF
chmod 600 .env
```

`TTN_TOKEN` needs at least read access to applications, devices and their
session keys. `TTN_STORAGE_APPLICATIONS` is a comma-separated list of TTN
application IDs to mirror via TTN Storage Integration.

### 3. Create the data directory

```bash
mkdir -p data decoders
```

### 4. Build images

```bash
docker compose build
```

### 5. Initialise the database

```bash
docker compose run --rm lorawan-tools python message_database.py
```

### 6. Fetch device sessions and generate decoder files

```bash
docker compose run --rm lorawan-tools python request_ttn_devices.py
docker compose run --rm lorawan-tools python device_database.py
```

This creates decoder files under `decoders/` on the host, which are mounted
into `lorawan-decoder` at runtime.

### 7. Run an initial TTN Storage fetch

```bash
docker compose run --rm lorawan-tools python ttn_storage_fetcher.py
```

This fetches up to 44 h of history for all configured applications.

### 8. Start the services

```bash
docker compose up -d
```

### 9. Set up cron jobs

Add to the host crontab (`crontab -e`), adjusting the path to `docker-compose.yml`:

```cron
*/20 * * * * cd /home/user/docker && docker compose run --rm lorawan-tools python ttn_storage_fetcher.py
0    * * * * cd /home/user/docker && docker compose run --rm lorawan-tools python reprocess_messages.py
0    2 * * * cd /home/user/docker && docker compose run --rm lorawan-tools sh -c "python request_ttn_devices.py && python device_database.py"
```

| Job | Interval | Purpose |
|---|---|---|
| `ttn_storage_fetcher.py` | every 20 min | fetch last 25 min of TTN Storage messages |
| `reprocess_messages.py` | every hour | retry decoding of messages received without a known session key |
| `request_ttn_devices.py` + `device_database.py` | daily at 02:00 | refresh device sessions and regenerate decoder files |

The 20-minute fetch passes `TTN_STORAGE_LAST=25m` implicitly via the
`lorawan-tools` environment and stays within TTN's rate limit of 10 requests
per 60 s.

## Running scripts manually

```bash
# Re-fetch device sessions
docker compose run --rm lorawan-tools python request_ttn_devices.py

# Regenerate decoder files
docker compose run --rm lorawan-tools python device_database.py

# Reprocess undecoded messages immediately
docker compose run --rm lorawan-tools python reprocess_messages.py

# Open a shell inside the tools container
docker compose run --rm lorawan-tools bash
```

The `--rm` flag removes the container after it exits.

## Updating

```bash
cd github/ttn-message-interceptor && git pull && cd ..
docker compose build
docker compose up -d
```

If the database schema changed, run the migration (idempotent):

```bash
docker compose run --rm lorawan-tools python message_database.py
```

---

## Notes

- `ttn_device_sessions.db` is not created automatically. Copy it from an
  existing Raspi deployment or let `request_ttn_devices.py` create it fresh.
- The `decoders/` directory is shared between `lorawan-decoder` (reads) and
  `lorawan-tools` (writes). No restart of `lorawan-decoder` is needed after
  decoder files are updated.
- `lorawan-tools` has `profiles: ["tools"]` and therefore does **not** start
  with `docker compose up`. It only runs when explicitly called via
  `docker compose run`.
